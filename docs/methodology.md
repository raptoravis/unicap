# 多 Agent 并行开发实操手册

> 生成日期：2026-04-25
> 方法论来源：atum.li《How to Scale AI Dev》（2026 春）—— 三把钥匙 + 四种并行模式
> 适用工具栈：Claude Code（Opus 4.7 / Sonnet 4.6）+ git worktree + Skill 库
> 示例项目：**程序化股票交易系统**（A 股 + 港股，日内 + 日线策略）
> 读者：希望从"1 个 Agent 写代码"升级到"3-6 个 Agent 协同产出"的工程师

---

## 〇、TL;DR — 一页能记住的东西

1. **瓶颈不在写代码，在你这个人**。瓶颈三连：需求传不清、正确性靠你审、架构靠你管。手册的全部目的就是把你从这三个单点上挪开。
2. **三把钥匙**：
   - **需求对齐** —— Agent 穷尽追问 + Top 3 方案，你只做选择题。
   - **TPDD（测试计划驱动开发）** —— 写代码前先把测试计划独立审掉，开发阶段 Agent 自己跑测试自己修。
   - **工程纪律 Skill 化** —— 把"模块深度 / 信息隐藏 / 命名 / 错误处理"编成 Skill，每个 Agent 启动时强制加载。
3. **四种并行模式**（按上手难度递增）：
   - 跨项目 → 同项目不同模块 → 同模块不同事务类型 → 单任务内部子模块。
4. **节奏**：需求对齐**串行**（装弹），开发执行**并行**（射击）。
5. **Agent 数量上限**：新手 3-4 个，熟练后 6-8 个。再多产出消化跟不上。
6. **路径不会更轻松**。你从"写代码的人"变成"运行协作系统的人"，脑力消耗反而上升，但产出量级提升。

## 一、为什么你现在 1 打 1 还累？—— 三个真实瓶颈

| 瓶颈 | 表现 | 单点是谁 |
|---|---|---|
| 需求传递 | "你写错了，我意思不是这个" 反复 3-5 轮 | 你的脑子 |
| 正确性验证 | 每个 PR 都要逐行读、跑一遍才敢合 | 你的眼睛 + 时间 |
| 架构纪律 | Agent 写得能跑，但风格、分层、命名各搞一套 | 你的 review |

只要这三件事都"必须你亲自做"，你开 N 个 Agent 也只是把瓶颈前移到自己面前，并不真正放大产出。

下面三章逐个解。

## 二、第一把钥匙：需求对齐（让 Agent 自己问，让你只做选择）

### 2.1 两个动作

**动作 A —— 穷尽式追问**：Agent 主动列出所有不确定，你逐条回答，直到剩下的问题都琐碎为止。

**动作 B —— Top 3 方案生成**：对每个仍然模糊的点，Agent 直接给出 3 个候选方案 + 各自的优缺点 + 推荐项。你只勾选。

**关键转变**：你不再是"回答问题的人"，你是"审方案的人"。

### 2.2 实操 Prompt 模板（直接可用）

```
我要做 X。在你写任何代码 / 测试计划之前，请只做两件事：

1. 列出你认为不确定、需要我确认的所有点。把它们分成：
   (a) 必须立刻确认（不定无法开工）
   (b) 可以先用合理默认值、稍后回调

2. 对 (a) 中的每一项，给出 Top 3 方案，每个方案标注：
   - 一句话描述
   - 预计实现复杂度（低/中/高）
   - 主要风险
   - 你的推荐项 + 理由（一句话）

不要写代码。不要给出超过这两点的内容。
```

**触发判据**：你大脑里只要有"我得想想这事我要怎么交代"的犹豫，就用这个模板，不要自己想清楚再去交代。

### 2.3 示例：交易系统的"风控模块"需求对齐

你最初的需求只是一句话：

> "做一个风控模块，限制持仓和回撤。"

Agent 用上面的 Prompt 跑完一轮，吐出来的（节选）：

**(a) 必须确认：**

| # | 不确定点 | 推荐 |
|---|---|---|
| 1 | 单票持仓上限是按金额、按净值百分比、还是按股本流通比？ | 净值百分比（A 股流通问题不在你 PoC 范围内） |
| 2 | 最大回撤是日内、滚动 N 日、还是从入金日累计？ | 日内 + 滚动 20 日双线 |
| 3 | 触发风控后是 **撤单 + 平仓** 还是 **撤单 + 冻结新单**？ | 撤单 + 冻结新单（避免日内反复砍仓滑点） |
| 4 | 风控判定在策略层、执行层还是独立服务？ | 独立服务（执行层和策略层都要拦截） |

**(b) 默认值即可：**
- 单票上限默认 5%、组合最大回撤默认 8% —— 都做成配置项。
- 时区按 Asia/Shanghai；港股开市单独配置。

你只做了 4 个选择题（约 90 秒），需求就锁死了。后续 Agent 不会再问"那回撤怎么算"。

### 2.4 反模式

- ❌ 你脑子里想清楚再来交代 —— 你想不清楚才需要 Agent 帮你穷举。
- ❌ Agent 问你"你觉得呢？" —— 这是失败的对齐，要求它必须给推荐项。
- ❌ 一次对齐 30 个问题 —— 拆成 5 个一组，避免你做选择疲劳。

## 三、第二把钥匙：TPDD —— 测试计划驱动开发

### 3.1 核心流程

```
需求锁定
   ↓
[Agent A] 生成测试计划（不写代码）
   ↓
[独立审核] —— 你 或 另一个空白会话 Agent
   ↓
测试计划锁定
   ↓
[Agent B] 写代码 + 自己跑测试 + 自己修 bug —— 直到全绿
   ↓
你只审：差异 + 是否有偷偷改测试计划
```

**为什么独立审核？** 因为生成测试计划的 Agent 和写代码的 Agent 共享上下文时，会下意识"为了让代码过测试而调整测试"。这一刀必须切开。

### 3.2 Agent 必须有"真操作环境"

否则测试不可信。按你的栈分别准备：

| 项目类型 | 必备工具链 |
|---|---|
| CLI / 后台服务 | shell + 完整日志 + pytest/jest/cargo test |
| Web 前端 | Playwright（已通过 MCP 接入 Claude Code）|
| GUI / 桌面 | 屏幕截图 + 控件树访问 |
| 复杂系统 | gdb / DAP 调试器 |
| 交易系统 | 历史 tick 回放 + 模拟券商网关 + 时间快进沙盒 |

**判据**：Agent 要能"看见"自己代码的运行结果。看不见就别相信"已通过测试"四个字。

### 3.3 测试计划生成 Prompt（直接可用）

```
针对 [需求文档 / 上一轮锁定的需求]，生成一份完整测试计划。要求：

1. 三层结构：
   - 单元测试：覆盖每个公开函数 / 类的正常 + 边界 + 异常路径
   - 集成测试：模块之间的契约
   - E2E：端到端的关键用户路径
2. 每条测试列出：
   - 测试 ID（可追溯）
   - 输入 / 前置条件
   - 期望输出 / 后置条件
   - 关联的需求条目
3. 标记 [P0] / [P1] / [P2]，P0 全过才允许合并。
4. 不要写实现代码。不要写测试框架代码。只列计划。

输出格式：Markdown 表格 + 每条一段简述。
```

### 3.4 示例：交易系统"风控模块"的测试计划（节选）

**单元层（P0）**

| ID | 输入 | 期望 |
|---|---|---|
| U-RISK-001 | 单票仓位 = 4.9% net | 通过 |
| U-RISK-002 | 单票仓位 = 5.0% net | 通过（边界包含）|
| U-RISK-003 | 单票仓位 = 5.01% net | 拒单 + 触发告警 |
| U-RISK-010 | 滚动 20 日回撤 = 7.99% | 通过 |
| U-RISK-011 | 滚动 20 日回撤 = 8.00% | 触发"冻结新单"状态 |
| U-RISK-020 | 时间在 09:25（A 股集合竞价）| 不应触发日内回撤判定 |

**集成层（P0）**

| ID | 场景 |
|---|---|
| I-RISK-001 | 策略层下单 → 风控层拦截 → 执行层不应收到该单 |
| I-RISK-002 | 持仓被市场行情波动推过 5% → 不主动平仓，但拒绝加仓 |
| I-RISK-003 | 风控状态从"冻结"恢复需手工置位，不可自动解除 |

**E2E（P0）**

| ID | 场景 |
|---|---|
| E-RISK-001 | 回放 2025-08-05 当天行情（暴跌日），验证 8% 回撤触发后所有新单被拒 |
| E-RISK-002 | 回放 2024-09-24（暴涨日），验证组合从满仓到 5% 单票限制不会反复刷新成交 |

**独立审核怎么做：**
- 你只看：是否覆盖了 §2.3 锁定的 4 个决策点？是否漏了边界值？
- 或者：开一个 **新的空白 Claude Code 会话**，喂入需求 + 测试计划，问："这份测试计划如果全过，能保证需求实现吗？漏了哪些？" 这个新会话没有写测试计划那个 Agent 的偏见。

### 3.5 开发阶段 Agent 的工作循环

测试计划锁定后，写代码的 Agent 进入这个循环（你不再介入）：

```
1. 实现 / 修改代码
2. 运行测试套件
3. 失败 → 看日志 → 定位 → 修 → 回到 2
4. 全绿 → 输出结构化摘要 → 等待你审 PR
```

**关键**：你审 PR 时不读全部代码，只读：

- diff 摘要（Agent 自动生成的"我改了什么 / 为什么"）
- 测试计划是否被修改（特别警惕悄悄删除 / 弱化测试用例）
- 关键架构决策点（参见第四把钥匙的 spot-check 列表）

## 四、第三把钥匙：工程纪律 Skill 化

把"好品味"编码成 Skill，每个 Agent 启动时强制加载，省得每次靠你 review 时一遍遍喊。

### 4.1 必备 Skill 清单（项目级，放 `.claude/skills/`）

| Skill 名 | 内容要点 |
|---|---|
| `module-depth` | 优先深模块（简单接口 / 复杂内部）；禁止"管道函数"式浅模块堆叠 |
| `info-hiding` | 模块间不暴露内部状态；公共类型只能是值对象，不能是行为承载者 |
| `error-policy` | 错误处理三选一：恢复 / 上抛 / 终止；禁止 try-catch 吞异常 |
| `naming` | 命名规则、领域术语表（参见 §4.3）|
| `comments-when` | 注释只写 "为什么"，不写 "做什么"；接口写不变量 / 边界条件 |
| `pr-summary` | 强制输出结构化摘要（参见 §7.2）|

启动命令固化在 `.claude/settings.json` 里，所有 Agent 自动继承。

### 4.2 你必须人工介入的 spot-check（无法 Skill 化）

| 场景 | 频率 |
|---|---|
| 新增跨模块接口 | 100% 审 |
| 新增数据库表 / schema 改动 | 100% 审 |
| 引入新依赖（`requirements.txt` / `Cargo.toml` 等）| 100% 审 |
| 删除测试用例 | 100% 审 |
| 单文件改动 > 300 行 | 100% 审 |
| 同模块内的内部重构 | 抽查 |

**理由**：这些是"长期投资"决策，错了未来你要还几个月利息，远超你 review 5 分钟的成本。

### 4.3 交易系统专属术语表（写进 `naming` Skill）

```
order        —— 用户提交的下单请求
trade        —— 撮合成交回报（不是 fill）
position     —— 当前持仓（净持仓，不分多空 leg）
exposure     —— 风险敞口（含未成交挂单）
nav          —— 组合净值
drawdown     —— 回撤（百分比，正数表示亏损）
session      —— 交易时段（9:30-11:30 / 13:00-15:00 / 港股不同）
```

强制：函数 / 变量 / 日志字段都用上述词，禁止 `qty / vol / hold` 之类同义词混用。Agent 一旦用错，Skill 会让它重写。

## 五、四种并行模式 —— 在交易系统中怎么落地

### 5.1 模式一：跨项目并行（最容易）

不同仓库不同 Agent。例如：

- Agent 1 在 `trading-engine` 仓库写撮合
- Agent 2 在 `factor-research` 仓库做因子回测
- Agent 3 在 `dashboard` 仓库做监控前端

各自跑各自的，几乎零协调成本。**第一周就该启用这个模式**。

### 5.2 模式二：同项目、不同模块（用 git worktree）

**条件**：模块文件重叠度 < 20%。

```bash
# 主仓库在 D:\dev\trading-engine，主分支 main
cd /d/dev/trading-engine

# 给每个并行任务开一个 worktree
rtk git worktree add ../trading-risk    feature/risk-module
rtk git worktree add ../trading-data    feature/data-feed
rtk git worktree add ../trading-exec    feature/exec-router
rtk git worktree add ../trading-bt      feature/backtest

# 每个 worktree 起一个独立 Claude Code 会话
# 在 ../trading-risk 目录下启动 Agent A
# 在 ../trading-data 目录下启动 Agent B
# ...
```

**交易系统的天然模块边界**（重叠度都很低）：

| 模块 | 主要文件 | 与其他模块的耦合面 |
|---|---|---|
| `data/` | tick / kline 订阅、历史回放 | 只通过事件总线发 `MarketEvent` |
| `strategy/` | 因子、信号、组合管理 | 订阅 `MarketEvent`，发 `OrderRequest` |
| `risk/` | 仓位 / 回撤 / 断路器 | 拦截 `OrderRequest`，输出 `OrderRequest` 或拒单 |
| `exec/` | 券商网关、撮合代理 | 接收 `OrderRequest`，发 `Trade` |
| `backtest/` | 撮合模拟、滑点模型 | 替换 `exec/` 的 backend |
| `monitor/` | dashboard、告警 | 只读所有事件总线消息 |

事件总线 + 6 个模块 = 6 个 worktree 可同时开工。

**冲突点只有一个**：事件类型定义（`events.py` / `events.rs`）。处理方式：

1. 第一周先**串行**做事件契约（一个 Agent 锁定后所有人共用）。
2. 锁定后写到 Skill 里，禁止任何模块 Agent 修改 `events.*`，必须通过 PR 申请。

### 5.3 模式三：同模块、不同事务类型

同一个"风控模块"内部，三个 Agent 同步开工：

| Agent | 任务 |
|---|---|
| Agent X | 写实现 + 单元测试 |
| Agent Y | 写集成测试 + E2E（基于 §3.4 的测试计划）|
| Agent Z | 修历史 issue（GitHub 上挂着的 8 个 risk 相关 bug）|

三者共享同一个 worktree 时容易踩脚 —— **建议各开一个 worktree、最后合并**。

### 5.4 模式四：单任务内部并行（前沿，谨慎用）

让一个 Agent 自己拆子任务再并行。例：让一个"实现风控模块"的 Agent 内部并行去做"持仓限制 / 回撤限制 / 断路器"三个子模块。

**目前状态**：Opus 4.7 已经能做，但稳定性不如外层并行；模块边界不够清晰时容易自相打架。

**建议节奏**：先把模式 1-3 跑顺 4 周以上，再尝试模式 4。

### 5.5 多路并试、择优合并（模式 2 的变体）

测试计划锁定后，**故意**让 2-3 个 worktree 用不同实现思路同时做风控模块：

- Worktree A：纯函数式 + 不可变状态机
- Worktree B：actor 模型 + tokio
- Worktree C：传统 OOP + 锁

跑同一份测试计划，比较：测试通过率、性能、可读性。**择优合并，其他 worktree 删掉**。

失败成本现在已经低到可以这么玩 —— 这是 atum 文章里强调的"心态变化"：从例外变成常态。

## 六、阶段四：测试 —— 交易系统不能只靠常规三层

常规三层（单元 / 集成 / E2E）是地板，不是天花板。交易系统至少再加三层：

### 6.1 历史回放测试（强制，P0）

选一组**已知会出问题的历史日**作为回放语料库，每次合并主干前必须回放通过：

| 日期 | 事件 | 检验点 |
|---|---|---|
| 2024-09-24 | A 股暴涨 / 涨停潮 | 单票限制不抖动、风控不误杀 |
| 2025-08-05 | 全球股灾 | 8% 回撤触发后新单全拒 |
| 2024-04-15 | 港股早盘熔断 | 跨市场状态机正确隔离 |
| 2025-04-07 | 关税新闻日内反转 | 滑点模型不爆 |
| 任意周一 09:25-09:30 | 集合竞价 | 风控不参与集合竞价价位计算 |

写成 Skill `replay-suite`，Agent 自动跑。

### 6.2 压力 + 混沌测试（P1）

- **行情风暴**：把历史 tick 回放速度调到 100x，看撮合 / 事件总线是否丢消息。
- **券商网关掉线**：随机 kill 网关进程 30 秒，看 `exec/` 模块的重连和未确认订单恢复。
- **时钟跳变**：把系统时间向前 / 向后跳 1 分钟，看 session 状态机会不会错。

### 6.3 资金安全回归（P0，每次合并都要跑）

| 不变量 | 检验方式 |
|---|---|
| 总资产 = 现金 + Σ(持仓 × 现价) | 每个事件循环结束后断言 |
| 成交回报数量 ≤ 下单数量 | 全 E2E 跑完后核对 |
| 任何 reject 路径不修改持仓 | 单元测试 + 故障注入 |
| 同一笔 order 不可被处理两次（幂等）| 集成测试 + 重放消息 |

这些是**资金不变量**，放进 `invariants` Skill，每个 Agent 修任何 `risk/` `exec/` 代码都必须重跑。

### 6.4 测试时长预算（参考）

| 层 | 单次跑完 | 何时跑 |
|---|---|---|
| 单元 | < 30s | 每次 Agent 自己跑（每次改代码）|
| 集成 | 1-3 min | 每次 Agent 完成一个子任务 |
| E2E + 回放 | 5-15 min | 合并 PR 前 + 每天定时一次 |
| 压力 / 混沌 | 30-60 min | 周维度，周五跑 |

如果你的项目压根没有"5 分钟内能跑完一组关键测试"的能力，先别开多 Agent —— 先建测试管道。

## 七、阶段五：合并与产出消化

这是 atum 明确指出的**未解决瓶颈**：Agent 产出快，你消化慢。两个抓手：

### 7.1 强制结构化 PR 摘要

每个 Agent 提 PR 时，必须由 `pr-summary` Skill 生成下面这份结构（不是自由文本）：

```markdown
## 改动范围
- 模块：risk/
- 文件数：7
- 行数：+412 / -23

## 决策点（要你审的）
1. [DECISION] 我把"冻结新单"实现成了独立的状态机，而不是 risk_state 上的字段。
```

### 7.2 合并节奏

| 项目 | 推荐 |
|---|---|
| PR 大小 | 单 PR 尽量 < 500 行 diff、< 7 个文件 |
| 合并频率 | 每天合 2-4 次主干，**避免周五合**（周末出问题没人盯）|
| 主干保护 | 主干必须过 P0 全套 + 回放套件 |
| Agent 任务粒度 | 1 任务 ≈ 1 模块的 1 个特性，平均 4-8 小时 Agent wall-clock |

### 7.3 你产能匹配公式（粗算）

```
你能消化的 Agent 数 ≈ (你每天能花在 review 上的小时数 × 60 分钟) / (每个 PR 的 review 时间)

举例：每天 4h review × 60 = 240 分钟
     每 PR 摘要 + spot-check 约 6 分钟
     → 上限 240 / 6 = 40 PR/天
     → 假设每 Agent 一天产出 5-8 PR
     → 上限 5-8 个 Agent
```

这个数和 atum 文章里说的"熟练 6-8 个"基本对得上。**Skill 越完善，单 PR review 时间越短，上限越高**。

## 九、常见陷阱

| 陷阱 | 现象 | 解法 |
|---|---|---|
| **跳过需求对齐** | "我先让它写写看" → 3 小时后整个推翻 | 强制 §2.2 模板，没跑过不准开 PR |
| **测试计划由写代码的 Agent 顺手生成** | PR 全绿，上线翻车 | §3.1 独立审核，新会话 Agent 二审 |
| **Skill 写成长篇大论** | Agent 不读 / 读不进去 | Skill < 200 字，例子 > 规则 |
| **Agent 越多越慢** | 5 个 Agent 你一个都顾不上 | 砍到 3 个，等 Skill 完善再加 |
| **多 Agent 改同一接口** | 合并地狱 | 把"接口改动"列入 §4.2 spot-check，集中串行处理 |
| **回放套件不维护** | 半年后再跑全挂 | 每出一次线上 incident，把当天行情加进 `replay-suite` |
| **资金不变量没写测试** | 偶发性多扣钱，难以复现 | §6.3，每次 PR 强制 |
| **Agent "我已经修复了"但其实没跑测试** | 提交了空 commit / 修改了断言 | PR 摘要里测试通过数必须 > 0 且不能下降 |

## 附录 C：判断"该不该上多 Agent"的三道闸

1. **测试 5 分钟内能跑完一组关键路径吗？** —— 否则 Agent 自己改自己跑的循环走不通。
2. **你能在 5 分钟内审完一个 PR 吗？** —— 否则 Agent 越多你越被淹。
3. **过去 1 个月有过"我说的不是这个"超过 3 次吗？** —— 是的话先去打磨 §2 的需求对齐，别加 Agent。

三道全过，再上多 Agent。任何一道没过，加 Agent 就是放大瓶颈。
